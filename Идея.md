````md
# App Design Spec: Time Manager with Mascot & Gamification

## 1. Базовая идея приложения

Мобильное приложение тайм-менеджмента, собранное на Nuxt 4 + Capacitor + Pinia, с локальным хранением данных в SQLite.

Главная цель приложения — быть простым ежедневным инструментом для задач, который не перегружает пользователя интерфейсом, но удерживает внимание за счёт лёгкой визуальной изменчивости, геймификации, маскота, уровней и инвентаря.

Приложение должно работать независимо от сервера. При этом архитектура заранее проектируется так, чтобы в будущем можно было заменить SQLite на REST API без переписывания UI и бизнес-логики.
````

# 4. Визуальный стиль приложения

## 4.1 Общий стиль

Приложение всегда работает в тёмной теме.

Светлая тема не нужна. Визуальная система строится вокруг тёмных фонов, крупных кнопок, читаемых карточек и мягких акцентных цветов.

Стиль:

```txt
тёмный
минималистичный
мобильный
игровой
мягкий
не перегруженный
с лёгкой визуальной вариативностью
```

Главное правило: структура интерфейса не меняется, меняется только визуальное ощущение.

Не меняются:

* расположение кнопки создания задачи;
* структура карточки задачи;
* положение списка задач;
* хедер;
* навигация;
* основной flow создания и выполнения задачи.

Могут меняться:

* оттенок фона;
* оттенок карточек;
* акцентный цвет;
* текст кнопки;
* заголовок списка задач;
* фон страницы пользователя;
* декоративные визуальные детали.

---

## 4.2 Зачем нужна визуальная изменчивость

Пользователь с СДВГ может быстро привыкать к однообразному интерфейсу и переставать его замечать.

Поэтому приложение должно сохранять стабильную структуру, но слегка менять визуальное оформление после входа в приложение или открытия страницы.

Идея:

```txt
структура стабильная → не надо заново разбираться
визуал чуть меняется → интерфейс не превращается в фоновый шум
```

Это не должна быть хаотичная смена дизайна. Меняется только контролируемый набор параметров.

# 5. Система тёмных визуальных вариантов

Цвет меняется по трём UI-осям:

```txt
1. appBackground — общий фон приложения
2. surfaceBackground — фон карточек и блоков
3. accentColor — кнопки, прогресс, выделения
```

Каждая ось может иметь пару тёмных цветов, между которыми выбирается случайное значение.

Пример:

```ts
export type ThemeAxis = {
  from: string
  to: string
}

export type ThemeRandomRange = {
  appBackground: ThemeAxis
  surfaceBackground: ThemeAxis
  accentColor: ThemeAxis
}
```

Пример конфига:

```ts
export const darkThemeRange: ThemeRandomRange = {
  appBackground: {
    from: "#0D0F14",
    to: "#111827"
  },
  surfaceBackground: {
    from: "#171923",
    to: "#1E1B2E"
  },
  accentColor: {
    from: "#7C5CFF",
    to: "#4DAA8A"
  }
}
```

Можно сделать не просто выбор одного из двух цветов, а генерацию промежуточного цвета между `from` и `to`.

# 6. Random visual helper

Нужен общий helper для визуальной рандомизации.

Он должен уметь:

* принимать ключ сущности;
* принимать список значений;
* принимать частоту обновления строкой;
* возвращать выбранное значение;
* опционально не повторять прошлое значение;
* сохранять результат в store или storage.

Частоты:

```ts
export type RandomFrequency =
  | "app-enter"
  | "page-enter"
  | "task-created"
  | "level-up"
  | "manual"
```

Описание:

```txt
app-enter     — обновлять при входе в приложение
page-enter    — обновлять при входе на страницу
task-created  — обновлять после создания задачи
level-up      — обновлять после повышения уровня
manual        — только вручную
```

Базовый helper:

```ts
export type PickRandomOptions<T> = {
  key: string
  values: T[]
  frequency: RandomFrequency
  previousValue?: T | null
  avoidRepeat?: boolean
}

export function pickRandomValue<T>(options: PickRandomOptions<T>): T {
  const { values, previousValue, avoidRepeat = true } = options

  if (!values.length) {
    throw new Error(`Random values list is empty for key: ${options.key}`)
  }

  if (values.length === 1) {
    return values[0]
  }

  let pool = values

  if (avoidRepeat && previousValue !== null && previousValue !== undefined) {
    pool = values.filter((value) => value !== previousValue)
  }

  const index = Math.floor(Math.random() * pool.length)

  return pool[index] ?? values[0]
}
```

# 11. Критичные правила UI

1. Главное действие всегда одно и всегда заметное.
2. Карточка задачи должна быть читаемой за 1–2 секунды.
3. Расположение элементов не должно прыгать.
4. Визуальная рандомизация не должна менять UX-сценарии.
5. Цвета не должны ломать контрастность.
6. Звук неудачи должен быть отключаемым.
7. Просроченные задачи должны подниматься вверх, но не превращать интерфейс в стрессовую помойку.
8. Завершённые задачи лучше сворачивать или уводить ниже.
9. Маскот и инвентарь не должны мешать созданию задач.
10. Геймификация должна усиливать основной flow, а не заменять его.

```

Главное уточнение по проектированию: визуальный random должен быть **не в компонентах**, а в отдельном слое. Компонент просто берёт `theme`, `buttonText`, `heading`, `background` из store. Тогда интерфейс останется управляемым, а не превратится в набор случайных `Math.random()` по всему приложению.
```
