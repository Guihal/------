# План разработки

## 1. Цель плана

Сначала собрать рабочее ядро задач. Маскот, инвентарь, уведомления и visual
random подключаются только после того, как создание и выполнение задач работают
стабильно.

Этапы разработки соответствуют [00-scope-map.md](00-scope-map.md).

## 2. MVP-0. Рабочее ядро

### Этап 0. Зафиксировать стартовые решения

Результат:

- принят scope MVP-0;
- подтвержден SQLite-плагин;
- подтвержден client-only Nuxt runtime;
- подтвержден подход к memory repositories для browser dev;
- подтвержден Vitest для доменных тестов.

Готово, если можно начинать scaffold без возврата к базовым спорам.

### Этап 1. Scaffold приложения

Сделать:

- создать Nuxt 4 проект;
- включить TypeScript;
- подключить Pinia;
- настроить базовую структуру папок;
- отключить SSR для мобильного runtime;
- добавить Capacitor;
- добавить Vitest;
- создать пустой app shell;
- проверить запуск в браузере.

Готово, если открывается пустой shell приложения с главным экраном.

### Этап 2. Core domain без UI

Сделать:

- типы `Task`, `Profile`, `Progression`;
- `suggestTaskComplexity`;
- `calculateTaskXp`;
- linear level progression с фиксированным `XP_PER_LEVEL`;
- ordered decision tree для сложности;
- `resolveTaskList`;
- правила создания задачи;
- правила выполнения задачи;
- правила архивации;
- unit-тесты для этих функций/use cases.

Готово, если доменная логика работает без Nuxt, Vue, Pinia и SQLite.

### Этап 3. Infrastructure MVP-0

Сделать:

- dependency container;
- `useAppDependencies`;
- SQLite database wrapper;
- `UnitOfWorkPort`;
- migration runner;
- `schema_migrations`;
- миграцию `001_initial`;
- `SqliteTaskRepository`;
- `SqliteProfileRepository`;
- `SqliteProgressionRepository`;
- memory repositories для browser dev;
- bootstrap профиля и progression.

Готово, если задача сохраняется локально в SQLite на Android и UI может
разрабатываться в браузере через memory repositories.

### Этап 4. Task flow

Сделать:

- главный экран;
- список задач;
- группы задач: просроченные, ближайшие, без дедлайна, выполненные;
- карточку задачи;
- форму создания;
- форму редактирования;
- выполнение задачи;
- архивацию задачи;
- отображение XP и уровня.

Готово, если пользователь может создать, увидеть, выполнить и архивировать
задачу, а данные сохраняются после перезапуска.

### Этап 5. MVP-0 polish

Сделать:

- empty states;
- loading states;
- error states;
- статичную темную базовую тему;
- базовые design tokens;
- проверку мобильных размеров;
- сборку Android;
- smoke-test на устройстве/эмуляторе.

Готово, если MVP-0 можно показывать как рабочий прототип без маскота и
уведомлений.

## 3. MVP-1. Маскот, инвентарь и XP-множители

### Этап 6. Inventory domain

Сделать:

- типы `Mascot`, `MascotSlot`, `InventoryItem`, `ItemStats`,
  `UserInventoryItem`, `EquippedItem`, `LevelReward`, `TaskRewardRoll`;
- правила выдачи предметов за уровни;
- правила task multiplier;
- правила XP-множителя надетых предметов;
- правила random drop после выполнения задачи;
- правила генерации XP-множителя предмета по редкости;
- защиту от повторной награды через `level_rewards`;
- защиту от повторного drop-roll через `task_reward_rolls`;
- unit-тесты выдачи наград, drop audit, source invariants, ownership checks и
  slot compatibility.

Готово, если награда за 5/10/15 уровень выдается один раз, а выполненная задача
не может получить повторный drop-roll.

### Этап 7. Inventory infrastructure

Сделать:

- миграцию для mascot/profile_mascot/mascot_slots/inventory/equipped/level_rewards/task_reward_rolls;
- seed каталога предметов;
- `InventoryRepository`;
- `MascotRepository`.
- memory repositories для mascot/inventory в browser dev.

Готово, если предметы и экипировка сохраняются локально.

### Этап 8. Profile, mascot, inventory UI

Сделать:

- экран профиля;
- отображение маскота;
- экран инвентаря;
- экипировку предметов;
- слой отображения предметов на маскоте;
- feedback получения предмета.

Готово, если пользователь видит полученный предмет и может надеть его.

## 5. Первый технический milestone

Минимальная цель первой рабочей версии:

```txt
Nuxt shell
dependency container
memory repositories for browser dev
SQLite on Android
schema migrations
главный экран
создание задачи
список задач
выполнение задачи
архивация задачи
XP
уровень
персистентность после перезапуска
```

Маскот, инвентарь, уведомления и visual random не блокируют этот milestone.

## 6. Что нельзя делать раньше времени

Не делать до рабочего MVP-0:

- маскота;
- инвентарь;
- экипировку;
- локальные уведомления;
- visual random;
- сложный профиль;
- много видов задач;
- сложные анимации;
- магазин предметов;
- повторяющиеся задачи;
- календарный экран;
- синхронизацию;
- REST API;
- сложную систему бонусов кроме XP-множителя предметов.

Причина: если задача не создается и не закрывается просто, остальные фичи не
спасут продукт.

## 7. Definition of Done

### MVP-0 готов, когда:

- приложение запускается без сервера;
- первый запуск создает локальный профиль;
- задачи сохраняются в SQLite;
- browser dev работает через memory repositories;
- migration runner применяет схему из пустой БД;
- активные задачи отображаются на главной;
- просроченные задачи видны выше обычных;
- задача создается с одним обязательным полем;
- сложность определяется автоматически и редактируется вручную;
- задача выполняется;
- повторное выполнение не начисляет XP второй раз;
- выполнение задачи, начисление XP и level-up выполняются транзакционно;
- задача архивируется вместо физического удаления;
- XP начисляется;
- уровень повышается через фиксированный `XP_PER_LEVEL`;
- данные сохраняются после перезапуска;
- UI не требует регистрации и настройки проектов.

### MVP-1 готов, когда:

- каталог предметов заполняется seed-данными;
- предмет выдается на 5-м уровне;
- повторная награда за тот же уровень невозможна;
- после выполнения задачи может выпасть предмет по XP-based drop chance;
- повторный drop-roll для одной задачи невозможен;
- надетый предмет применяет XP-множитель к будущим задачам;
- профиль показывает маскота;
- инвентарь показывает полученные предметы;
- предмет можно надеть и снять;
- экипировка сохраняется после перезапуска.

### MVP-2 готов, когда:

- локальные уведомления работают для задач с дедлайном;
- уведомления отменяются при выполнении/архивации;
- notification reconciliation чинит расхождения после перезапуска;
- visual random работает только на заданных событиях;
- visual state сохраняется;
- настройки позволяют отключить visual random/reduced motion;
- UI проходит базовую проверку контрастности и мобильных размеров.
