# Видение приложения

## 1. Рабочее определение

Приложение - мобильный тайм-менеджер для повседневных задач, рассчитанный на
пользователей, которым трудно долго удерживать внимание, поддерживать сложную
систему планирования и регулярно возвращаться в перегруженные таск-менеджеры.

Рабочее название в документах: `Task Companion`. Название не является финальным
брендом.

## 2. Главная идея

Пользователь должен быстро открыть приложение, увидеть актуальные задачи,
добавить новую задачу без настройки системы и получить понятную обратную связь
за выполнение.

Приложение не пытается заменить Notion, календарь, CRM или сложный GTD-комбайн.
Оно решает более узкую задачу: удержать ежедневный список дел живым, простым и
достаточно приятным, чтобы пользователь возвращался к нему.

## 3. Целевая аудитория

Основная аудитория:

- студенты и молодые специалисты;
- пользователи, которым сложно пользоваться перегруженными планировщиками;
- пользователи с трудностями концентрации внимания;
- люди, которым нужен личный список задач без командной работы и настройки
  рабочих пространств.

Приложение не позиционируется как медицинский инструмент и не ставит диагнозы.
Корректная формулировка: поддержка пользователей с трудностями концентрации и
повышенной когнитивной нагрузкой.

## 4. Проблема пользователя

Существующие решения часто требуют от пользователя:

- заранее настроить проекты, списки, теги, доски или шаблоны;
- держать в голове структуру приложения;
- переключаться между несколькими экранами;
- вводить слишком много полей при создании задачи;
- самостоятельно поддерживать мотивацию без быстрой обратной связи.

Для целевой аудитории это приводит к тому, что приложение сначала кажется
полезным, но быстро превращается в еще одну незавершенную систему.

## 5. Продуктовое обещание

`Task Companion` помогает быстро фиксировать и закрывать задачи без лишней
организации. После появления базового ядра приложение усиливается легкой
геймификацией, маскотом и контролируемой визуальной изменчивостью.

Ключевой принцип:

```txt
структура стабильная, ощущение слегка меняется
```

Пользователь не должен каждый раз заново учиться интерфейсу, но приложение не
должно становиться полностью незаметным фоновым списком.

## 6. Принципы продукта

1. Один основной экран для повседневной работы.
2. Одно главное действие - создать задачу.
3. Создание задачи без обязательной настройки.
4. Актуальные активные задачи видны на главной без перехода в проекты.
5. Offline-first: основные функции работают без сети.
6. Геймификация поддерживает выполнение задач, но не заменяет его.
7. Маскот и инвентарь не мешают задаче и не конкурируют с главным действием.
8. Никаких наказаний, которые усиливают стресс.
9. Визуальная рандомизация не меняет сценарии и расположение элементов.
10. Пользователь всегда может понять, что сейчас важно сделать.

## 7. Scope по этапам

Каноническое разделение этапов описано в [00-scope-map.md](00-scope-map.md).

Коротко:

- `MVP-0` - рабочее ядро задач, SQLite, XP и уровни;
- `MVP-1` - маскот, инвентарь, XP-множители и награды;
- `MVP-2` - локальные уведомления, visual random и настройки;
- `Post-MVP` - сервер, синхронизация, команды, календарь, привычки, Pomodoro и
  остальные расширения.

## 8. MVP-0

Первая версия должна доказать, что task flow работает.

Входит:

- главный экран со списком актуальных задач;
- быстрое создание задачи;
- редактирование задачи;
- выполнение задачи;
- архивация задачи;
- автоматическое предположение сложности задачи;
- ручное изменение сложности;
- приоритет задачи;
- дата/время дедлайна;
- локальное хранение в SQLite;
- локальный профиль пользователя;
- уровень и общий прогресс;
- статичная темная базовая тема без visual random.

MVP-0 считается успешным, если пользователь может закрыть приложение, открыть
его снова и продолжить работу с теми же задачами.

## 9. MVP-1

После рабочего ядра добавляется слой мотивации.

Входит:

- полноценный экран пользователя;
- базовый маскот;
- выдача предметов за уровни;
- редкая случайная выдача предметов после выполнения задачи;
- ролл XP-множителя при получении предмета;
- применение XP-множителей от надетых предметов;
- инвентарь;
- экипировка предметов на маскота;
- отображение полученных предметов;
- защита от повторной выдачи награды за один и тот же уровень или задачу.

В MVP-1 единственная характеристика предмета - бонус к XP. Предметы не меняют
сложность задач, шанс выпадения других предметов и не вводят штрафы.

## 10. MVP-2

После ядра, инвентаря и XP-множителей добавляются удержание внимания и настройки.

Входит:

- локальные напоминания для задач с дедлайном;
- request permissions;
- отмена уведомлений при выполнении/архивации;
- пересоздание уведомлений при изменении дедлайна;
- контролируемая визуальная вариативность;
- варианты текста главной кнопки;
- варианты заголовка списка;
- фон профиля;
- настройки;
- `reducedMotion`;
- `disableVisualRandomness`.

## 11. Не входит в MVP

Не делаем до завершения MVP-0/1/2:

- регистрацию и авторизацию;
- синхронизацию между устройствами;
- сервер;
- командные задачи;
- комментарии;
- вложения файлов;
- календарь как отдельный полноценный модуль;
- канбан-доски;
- привычки;
- Pomodoro;
- сложные повторяющиеся задачи;
- магазин предметов;
- донат;
- социальные функции;
- ИИ-помощника;
- медицинские рекомендации;
- предметы с бонусами к шансам выпадения, штрафам или другим механикам кроме
  XP-множителя.

Эти функции сохраняются как Post-MVP направления, но не блокируют приложение.

## 12. Основные пользовательские сценарии

### 12.1 Первый запуск

1. Пользователь открывает приложение.
2. Видит главный экран без регистрации.
3. Может сразу создать задачу.
4. Локальный профиль создается автоматически.
5. Прогресс начинается с первого уровня.

### 12.2 Создание задачи

1. Пользователь нажимает главную кнопку.
2. Вводит только название.
3. При желании указывает дедлайн, приоритет, описание.
4. Сложность определяется автоматически как предположение.
5. Пользователь может изменить сложность вручную.
6. После сохранения возвращается на главный экран.
7. Задача появляется в списке.

Обязательное поле только одно: название.

### 12.3 Выполнение задачи

1. Пользователь нажимает отметку выполнения в карточке.
2. Задача получает статус `completed`.
3. Пользователь получает XP.
   В MVP-0 это `finalXp = baseXp`.
   В MVP-1 это `finalXp = round(baseXp * taskMultiplier * equipmentXpMultiplier)`.
4. Если набран новый уровень, показывается спокойный level-up feedback.
5. В MVP-1 при достижении нужного уровня может выдаться предмет.
6. В MVP-1 после выполнения задачи может сработать random drop; шанс зависит от
   итогового XP за задачу.
7. Выполненная задача уходит ниже или в свернутый блок.

### 12.4 Просрочка задачи

1. Если дедлайн прошел, задача получает визуальный признак просрочки.
2. Просроченные задачи поднимаются выше обычных.
3. Приложение не отнимает опыт и не использует жесткую негативную лексику.
4. Пользователь может выполнить, перенести или архивировать задачу.

Термин в интерфейсе: `просрочено`, не `провалено`.

### 12.5 Напоминания

Сценарий относится к MVP-2.

1. Если у задачи есть дедлайн, напоминание может быть включено.
2. Время напоминания по умолчанию - за 30 минут до дедлайна.
3. Если дедлайн ближе чем через 30 минут, напоминание ставится на момент
   дедлайна.
4. Если дедлайна нет, точное локальное уведомление не создается.
5. Глобальные мягкие ежедневные напоминания остаются Post-MVP.

### 12.6 Маскот и инвентарь

Сценарий относится к MVP-1.

1. Пользователь открывает профиль.
2. Видит уровень, прогресс и маскота.
3. При получении предмета предмет попадает в инвентарь.
4. Пользователь может надеть предмет на маскота.
5. Надетые предметы дают XP-множитель для будущих задач.

## 13. Ключевые экраны

### Главная

Содержит в MVP-0:

- header с кратким статусом;
- крупную кнопку создания задачи;
- заголовок списка;
- группу просроченных задач;
- группу ближайших задач с дедлайном;
- группу задач без дедлайна;
- свернутый блок выполненных задач.

Главная не должна превращаться в dashboard. Если задач много, интерфейс должен
группировать их, а не вываливать один бесконечный список без структуры.

### Создание/редактирование задачи

Поля:

- название;
- дедлайн;
- приоритет;
- сложность;
- описание;
- переключатель напоминания в MVP-2.

Первый экран формы должен быть коротким. Дополнительные поля можно визуально
отделить как вторичные.

### Профиль

MVP-0:

- имя пользователя;
- уровень;
- прогресс до следующего уровня.

MVP-1:

- маскот;
- краткая статистика;
- переход в инвентарь.

MVP-2:

- настройки.

### Инвентарь

Экран относится к MVP-1.

Содержит:

- список полученных предметов;
- редкость;
- тип слота;
- состояние экипировки;
- действие надеть/снять.

## 14. Геймификация

Геймификация должна быть линейной и предсказуемой:

- XP начисляется за выполнение задач;
- уровень растет линейно: фиксированное количество XP на каждый уровень;
- предметы выдаются на каждом 5-м уровне в MVP-1;
- предметы могут редко выпадать после выполнения задачи, шанс зависит от XP;
- надетые предметы могут давать XP-множитель;
- характеристики предмета роллятся при получении, пока только XP-множитель;
- нет штрафов за просрочку;
- нет ежедневных серий как обязательной механики.

Цель геймификации - дать легкое ощущение прогресса, а не создать вторую систему
обязанностей.

## 15. Визуальная вариативность

Фича относится к MVP-2.

Разрешено менять:

- оттенок фона;
- оттенок карточек;
- акцентный цвет;
- текст главной кнопки;
- заголовок списка;
- фон профиля;
- мелкие декоративные детали.

Запрещено менять:

- расположение главной кнопки;
- структуру карточки задачи;
- место списка задач;
- основной порядок действий;
- семантику цветов статусов;
- доступность и контрастность.

## 16. Критерии успеха

### MVP-0 готов, если пользователь может:

- запустить приложение без сервера;
- создать задачу за 1-2 обязательных действия;
- увидеть актуальные активные задачи на главной;
- выполнить задачу;
- получить XP;
- поднять уровень;
- закрыть и снова открыть приложение без потери данных.

### MVP-1 готов, если пользователь может:

- открыть профиль с маскотом;
- получить предмет за уровень;
- получить редкий предмет после выполнения задачи;
- открыть инвентарь;
- надеть предмет на маскота.

### MVP-2 готов, если пользователь может:

- включить локальное уведомление для задачи с дедлайном;
- увидеть контролируемое изменение визуала;
- отключить visual random/reduced motion в настройках.

## 17. Главные продуктовые риски

1. Геймификация может начать мешать управлению задачами.
2. Визуальная рандомизация может разрушить ощущение стабильности.
3. Архитектура может стать тяжелее самого MVP.
4. Автоматическая сложность может раздражать, если пользователь не может ее
   поправить.
5. Напоминания могут стать шумом, если включать их без понятного времени.

Решение: держать MVP-0 узким, делать правила явными, а декоративные механики
подключать только после рабочего task flow.
