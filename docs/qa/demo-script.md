# Демо-сценарий (защита)

Буквальный чеклист для живого показа. 10 шагов из `docs/packets/quality-demo-acceptance.md`
и `docs/10-rebuild-technical-spec.md` §17.1. Выполнять по порядку, отмечать
галочкой. Не проза — экшены и что проверять глазами.

## Подготовка (один раз, до защиты)

- [ ] `cd backend && DATABASE_URL=... go run ./cmd/migrate` — миграции на пустую БД.
- [ ] Проверить/заменить фейковый bcrypt-хэш в `backend/seed/demo.sql` (строка с
      `admin@example.test`, `$demo$change-me-admin-password-hash`) на реальный
      хэш известного пароля, либо пропустить seed и создать админа вручную (см. шаг 9).
- [ ] `go run ./cmd/seed` — накатить demo seed (категории, mascot, каталог предметов).
- [ ] `cd backend && go run ./cmd/server` — backend на `http://127.0.0.1:8080`.
- [ ] `cd apps/mobile && npm run dev` — mobile на `http://127.0.0.1:3000`.
- [ ] `cd apps/admin && npm run dev -- --port 3001` — admin на `http://127.0.0.1:3001`
      (порт отдельный, чтобы не конфликтовать с mobile).
- [ ] Открыть mobile в браузере на 320–390px viewport (DevTools device toolbar).

## Шаги demo path

**1. Регистрация нового пользователя** — mobile, `/register`.
Ввести email/пароль/имя → `Создать аккаунт`. Ожидать: редирект на onboarding
или `/tasks`, без ошибки занятого email.

**2. Создание первой задачи только по названию** — mobile, `/tasks` → форма
создания. Ввести только `Название` (например "Помыть посуду"), `Сохранить`.
Ожидать: задача появляется в активном списке без полной перезагрузки страницы,
priority=`normal`, complexity=`medium`, категория=`общее`.

**3. Выполнение задачи** — mobile, карточка задачи → `Выполнить` (или
детали задачи → `Выполнить`). Ожидать: кнопка блокируется на время запроса,
повторный тап не создает вторую награду.

**4. Показ XP / multiplier / reward или no-drop** — mobile, reward popup после
шага 3. Ожидать: `+N XP`, если есть multiplier — `x1.25`/`x1.5`/`x2.00`
человекочитаемо, если предмет выпал — название/редкость/слот/xp-multiplier,
если нет — компактный фидбек `+N XP · награда не выпала`. `roll_value`
на экране быть не должно.

**5. Просмотр профиля** — mobile, `/profile`. Ожидать: имя, уровень, total XP,
прогресс до след. уровня, статистика (создано/выполнено/архивировано),
маскот с текущей экипировкой.

**6. Открытие инвентаря** — mobile, `/inventory`. Ожидать: список предметов
пользователя, фильтры по слоту/редкости, preview маскота.

**7. Экипировка предмета (если доступен)** — mobile, `/inventory` → карточка
предмета → `Надеть`. Если слот занят — UI явно показывает замену. Ожидать:
статус карточки меняется на "надет", маскот-preview обновляется.

**8. Другой пользователь не видит эти данные** — открыть второй
браузер/инкогнито-окно, mobile `/register` → зарегистрировать второго
пользователя. Ожидать: `/tasks` пуст (нет задачи из шага 2), `/inventory` пуст
(нет предмета из шага 7). При желании показать в DevTools Network, что прямой
запрос `GET /tasks/{id из шага 2}` вторым пользователем вернет `404`.

**9. Вход в админ-панель** — admin, `/login`. Если seed-хэш не заменен на
реальный: `psql $DATABASE_URL -c "UPDATE users SET password_hash =
'<bcrypt-хэш известного пароля>' WHERE email='admin@example.test';"` заранее.
Ввести email/пароль админа → `Войти`. Ожидать: редирект на `/` (dashboard),
обычный пользователь с тем же flow получает
`У этой учетной записи нет доступа к админ-панели`.

**10. Просмотр users/items/audit** — admin:
- [ ] `/users` — таблица пользователей, виден пользователь из шага 1 и шага 8,
  поиск по email работает.
- [ ] `/items` — каталог предметов, виден предмет, выданный на шаге 7 (по
  названию из seed-каталога).
- [ ] `/logs` — audit log, видны события из шагов 1–7: `auth.register`,
  `auth.login_success`, `reward.complete`, `reward.roll`, `inventory.equip`
  (и `reward.level`, если был level-up), `auth.admin_login` из шага 9.

## Что делать при сбое

- Если backend требует ручной правки БД для восстановления demo-состояния —
  это повод остановиться и эскалировать (см. packet Escalation), а не чинить
  вручную во время показа.
- Держать запасной seed-снапшот БД (`pg_dump`) на случай, если demo-прогон
  испортил состояние прямо перед защитой.
